Doel van het project

Het onderzoek doen naar, experimenteren met, en implementeren van een systeem dat een nieuwe user interaction experience biedt en uitbreidbaar en interactief dient te zijn.

Conceptbeschrijving

We gaan een interactieve spiegel maken voor in de badkamer. Je kunt op verschillende ‘innovatieve’ manieren interacteren met deze spiegel. Denk aan facial recognition, gesture control e.d. De spiegel zelf is niet interactief, maar extra hardware maakt dat wel mogelijk. Er zal een camera bij de spiegel komen om alle visuele input te verwerken, een microfoon om spraak input te verwerken en eventueel nog andere sensoren om andere input te verwerken. In/achter de spiegel zit een scherm waarop beeld getoond kan worden. De spiegel is van een speciaal materiaal gemaakt dat reflecteerd zoals normale spiegels, maar ook het beeld door laat van het scherm er achter. Deze spiegel fungeert als hub, d.w.z. er komt vanalles samen. Er zijn verschillende vormen van input zoals hierboven al beschreven en er gaan ook verschillende manieren van output komen, in de vorm van apps. Er kan bijvoorbeeld een app voor worden gemaakt die het weer laat zien, of een muziek speler, of een agenda, in principe alles wat je maar kan bedenken. Elke app kan zijn eigen input en output vormen hebben. Toekomstige uitbreidingen op het concept zouden kunnen zijn dat de spiegel samenwerkt met andere ‘slimme’ apparaten in huis (interactieve koelkast, tafel, bed, etc). Denk aan: terwijl je je tanden staat te poetsen start je een app om het nieuws te kijken, en dat je daarna naar de woonkamer loopt, en op een interactieve tafel het nieuws verder kijkt.

Werkwijze

Elke dag dat er aan het project gewerkt wordt, start met een scrum standup. Hier wordt per persoon besproken wat hij afgelopen dag heeft gedaan, hoe dat ging (problemen tegengekomen?) en wat hij die dag zelf gaat doen. Zo weet iedereen altijd van iedereen waar hij mee bezig is, waar problemen zich voordoen en wat de algemene voortgang van het project is. Elke week op woensdagmiddag wordt aan de klant getoond wat de status van het project op dat moment is: nieuwe functionaliteiten worden gedemonstreerd. Daarna wordt een nieuwe scrum sprint gestart: in overleg met de klant wordt bekeken wat het beste lijkt om de komende sprint tijd aan te gaan besteden en wat te gaan implementeren. De projectgroep geeft dan advies aan de klant m.b.t. wat technisch haalbaar is etc. Zo krijgt de klant elke week een update van de huidige staat van het project en weet hij waar de komende week zijn geld aan wordt gespendeerd.

Toewijzing van taken

Voor het verdelen en toewijzen van taken maken we gebruik van Redmine. Redmine is een online project management tool welke we gebruiken voor scrum en alle andere project management gerelateerde taken.

Versiebeheer

Voor versiebeheer gebruiken we Git + Gerrit. Gerrit is een online review tool die kwaliteit van de code kan bewaken en waarborgen. Gerrit houdt nieuwe commits als het ware tegen; deze worden pas in de remote repository gemerged op het moment dat iemand anders de commit bekeken, beoordeeld en goedgekeurd heeft. Elke taak zoals die in Redmine gedefinieerd wordt zal in zijn eigen commit terug te vinden zijn in de commit history, inclusief details zoals wie de commit heeft gemaakt en wanneer.

Testen

Mirror interacties (C#)

We gebruiken veel functies uit de library van de Kinect 2.0. Deze zijn al intern getest en we gaan dus niet opnieuw de functionaliteit hiervan testen met unittests o.i.d. Wel gaan we unittests schrijven voor onze eigen classes en functies. We gebruiken hiervoor het interne .NET unittest framework zodat we geen rompslomp hebben met externe libraries.

Mirror server (Javascript & node.js)

Ook hier maken we veel gebruik van libraries die al intern getest zijn. We hebben relatief weinig code nodig om het aan elkaar te knopen zodat het geheel functioneel is, maar de code die we hebben kunnen we wel testen. De server is als het ware een frontend en backend in een, en moet dan ook apart van elkaar getest worden, dat is wel vervelend want daar hebben we ook aparte test frameworks voor nodig: jasmine-node voor de nodejs backend en jasmine voor de plain javascript frontend.

Testresultaten

Bij elke nieuwe commit push worden alle tests van het betreffende project gedraaid op de jenkins build server. Als een test faalt, of de warning threshold wordt overschreden, dan wordt de commit afgekeurd en moet de commit aangepast en opnieuw gepushed worden zodat deze opnieuw getest kan worden. De commit wordt pas gereviewed als hij goedkeuring heeft van jenkins.
